Vitest OOM Crash on Node 18
FATAL ERROR: Zone Allocation failed — process out of memory
Config Bug Fixed
Symptom
FATAL ERROR: Zone Allocation failed — process out of memory when running npm test
Root Cause
Vitest's default threads pool combined with jsdom uses too much memory on Node 18's 512 MB default heap. Each test worker spawns a full V8 heap.
Fix Applied
Fix 1: Added pool: 'forks' to vitest.config.ts — forks use separate processes with lower overhead.

Fix 2: Changed test script in package.json to invoke Vitest with an explicit 4 GB heap cap.
vitest.config.ts vitest.config.ts 
pool: 'forks',
isolate: true,
package.json test script package.json 
"test": "node --max-old-space-size=4096 node_modules/vitest/vitest.mjs run"

        'vendor-react':  ['react', 'react-dom'],
      },
    },
  },

429 RPM vs RPD Error Classification Bug
Rate-limit errors (retryable) classified as quota exhausted (not retryable)
Bug Fixed
Symptom
Rate-limit (RPM) errors — which should trigger a retry — were instead treated as daily quota (RPD) exhausted, stopping transcription entirely.
Root Cause
lower.includes('exhausted') matched resource_exhausted, which is the Gemini SDK error code for all 429 responses — not just daily quota ones. This caused false non-retryable classification.
Fix Applied
Removed 'exhausted' from the RPD check. Only messages containing 'quota', 'daily', or 'per day' now trigger the non-retryable path. All other 429s remain retryable.
Before (broken) gemini.ts 
// BUG: 'exhausted' matches resource_exhausted (ALL 429s)
const isRpd = lower.includes('exhausted')
            || lower.includes('quota')
            || lower.includes('daily');
After (fixed)
// FIX: Only real daily-quota messages are non-retryable
const isRpd = lower.includes('quota')
            || lower.includes('daily')
            || lower.includes('per day');

buildTranscriptionPrompt Not Filtering Empty Participants
Empty strings leaked into the prompt; test "filters out empty strings" failed
Bug Test Fixed
Symptom
Test "filters out empty strings from participants array" failed. Empty strings like "" were appearing verbatim in the prompt sent to Gemini.
Root Cause
The participant line used .map() directly on the raw participants array without a prior .filter() step to remove empty or whitespace-only entries.
Fix Applied
Added .filter(p => p.trim()) before .map() and wrapped each name in quotes to prevent sentence-structure issues in the prompt.
Fixed participant line gemini.ts 
ctx!.participants
  .filter(p => p.trim())          // drop empty / whitespace-only
  .map(p => `"${p.trim()}"`)        // wrap in quotes
  .join(', ')

Groq Hallucinating Participant Names During Silence
Repeated lines like "[00:01:18] Participants, Harish, Murali." every 30 s
Bug Fixed
Symptom
Transcript output contained repeated lines like [00:01:18] Participants, Harish, Murali. appearing every ~30 seconds during silent gaps.
Root Cause
Passing "Meeting participants: Harish, Murali." as the Whisper prompt caused Whisper to echo speech-like sentences verbatim during silence — a known Whisper hallucination behavior when the prompt resembles a sentence.
Fix Applied
Fix 1: Changed prompt format from a sentence "Meeting participants: X, Y." to bare vocabulary hints "X, Y" — bare words are treated as spelling hints, not speech to repeat.

Fix 2: Added a post-processing regex filter to strip any output line that accidentally matches the participant name pattern.
Prompt format change groq.ts openai.ts 
// Before (sentence format — Whisper echoes it):
prompt: `Meeting participants: ${names}.`

// After (bare vocabulary hints — safe):
prompt: names.join(', ')  // e.g. "Harish, Murali"
Expected behavior note: No speaker labels (Speaker A/B) with Groq is intentional — Whisper does not support diarization. Speaker labels are a Gemini-only feature.

Render Build Fails — vite: Permission Denied
sh: 1: vite: Permission denied after npm install succeeds
Deployment Bug Fixed
Symptom
Render build log: sh: 1: vite: Permission denied after npm install completes successfully. npx vite build also fails with the same error.
Root Cause
Windows git does not preserve Unix execute (+x) bits. When the repo is cloned on Render's Linux server, node_modules/.bin/vite shell script loses its execute permission. npx also fails because it invokes the same binary.
Fix Applied
Changed the build script from "vite build" to "node ./node_modules/vite/bin/vite.js build" — invokes Node directly on the JS entrypoint, bypassing shell script execute permissions entirely. Render build command reverted to npm install && npm run build.
package.json build script fix package.json 
// Before (breaks on Render Linux):
"build": "vite build"

// After (calls Node directly — no exec bit needed):
"build": "node ./node_modules/vite/bin/vite.js build"
This is a cross-platform gotcha unique to developers on Windows who push to Linux CI/CD servers. The fix is permanent — no chmod required.
